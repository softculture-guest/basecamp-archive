✨ Направления и проекты в Adesk

В Adesk существует две категории, которые нам подходят: направления и проекты. Один проект может принадлежать только к одному направлению, то есть он как бы вложен в него.

Есть два варианта, как классифицировать расходы по этому принципу. Первый вариант — делить направления на очень верхнем уровне:
  • B2C
    • Очные курсы и воркшопы
    • Дист. курсы и онлайн-курсы
    • Марафоны
    • Видеокурсы
  • B2B
    • Корп. курсы
    • (Корп. видеокурсы)
    • Консалтинг и другие услуги
  • Прочие
    • Коворкинг
Это позволит нам понять, как работает каждое из крупных направлений бизнеса по отдельности, и как работает каждый из продуктов. Однако недостатком этой схемы будет отсутствие на уровне управленческой отчетности группировки по каждому конкретному курсу или программе.

Отчетность, сделанная по такому принципу будет выглядеть примерно вот так. Этот вариант соответствует логике планирования доходов исходя из расчетных показателей, а также бизнес-моделе в целом (правда бизнес-модель конечно не считается по completed contract method):

Второй вариант — классифицировать по конкретным продуктам или группам продуктов:
  • Очные курсы и воркшопы
    • PSH
    • IND
    • ...
  • Дист. курсы и онлайн-курсы
    • PSH
    • PRT
    • ...
  • Марафоны
    • MRM
    • MRF
    • MKS
    • ARO
  • Корп. курсы
    • CAP
    • CMT
  • Консалтинг
  • Коворкинг
Здесь мы будем видеть конкретные группы курсов, однако их список будет лишь бесконечно расти, что в какой-то момент сделает отчетность не очень операбельной, а синхронизацию довольно сложной вот почему:
  1. надо следить за обновлением шифров во всех местах, если они обновятся в базе курсов (особенно проблемным выглядит обновление шифра постфактум);
  2. надо отдельно разрабатывать механизм синхронизации проектов (то есть групп курсов или аббревиатур), так как для каждого нового шифра проект в Adesk придется создавать «на лету» и фиксировать где-то его идентификатор. При этом в базе данные хранятся не по аббревиатурам, а по полным шифрам, соответственно там будет очень сложно хранить идентификатор проекта Adesk (структуры данных не совпадают);
  3. еще сложнее будет синхронизировать доходы — вместо добавления доходов по каждому «счету» и типу продуктов, надо будет добавлять доходы по каждой группе курсов и «счету» — 100—150 позиций вместо 10—15 ежемесячно. При этом непонятно, где будет храниться идентификатор Adesk для проектов (аббревиатур), видимо надо также разрабатывать механизм их синхронизации;
  4. количество проектов будет только расти, особенно из-за марафонов. Уже сейчас проектов 124 для всех курсов СК, и 67 для курсов 2020-го и 2021-го. Анализировать сотню позиций в таблице — довольно странное занятие.
Вопрос пока такой: насколько нам нужна на уровне управленческой отчетности деление по группам продуктов, если есть продуктовая отчетность, которая прекрасно работает именно с отображением доходов по продуктам и группам продуктов? И насколько управленческая отчетность, построенная по принципам управленческого учета (completed contract method) вообще корректно использовать для этой задачи?
